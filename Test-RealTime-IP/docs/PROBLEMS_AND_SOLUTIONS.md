# المشاكل اللي قابلتنا وإزاي حلّيناها

> سجلّ بالمشاكل الحقيقية اللي ظهرت أثناء تطوير نظام التشارت (لحظي وتاريخي) وحلولها الفنية.

---

## 1. الـ History بطيء جداً ومش بيرجّع البيانات

**العرَض:** طلب history لفترة طويلة (مثلاً سنة+) بيعلّق السيرفر ومبيرجّعش داتا.

**السبب الجذري:** الـ `/history` endpoint كان بيلفّ على الـ stored proc **صفحة صفحة** (1000 صف كل مرة) باستخدام `OFFSET ... FETCH`. مع ملايين الصفوف ده بيعمل:
- آلاف الـ round-trips لقاعدة البيانات.
- سلوك **O(n²)** لأن الـ `OFFSET` بيتخطّى الصفوف من الأول في كل صفحة.

**الحل:**
- في `ApiServer.js`: استبدلنا اللوب بـ **query واحدة** ترجّع كل الصفوف مرة واحدة (`page_size = MAX_RAW_ROWS`).
- الاعتماد على الـ index `IX_HistorianData_TagId (tag_id, timestamp DESC)` لاستعلام سريع.

**النتيجة:** الاستعلام بقى ثواني بدل تعليق/timeout.

## 2. مستحيل نرسم ملايين النقط في Chart.js

**العرَض:** حتى لو جبنا الداتا، رسم مليون+ نقطة بيجمّد المتصفح.

**الحقيقة الفنية:** الشاشة عرضها ~1500 بكسل — فيزيائياً مستحيل تعرض أكتر من ده من النقط المنفصلة.

**الحل:**
- **Decimation عند العرض فقط:** نخزّن كل الخام في الرام، لكن نرسم **حد أقصى `MAX_DISPLAY_POINTS` (3000)**.
- استخدمنا خوارزمية **LTTB (Largest Triangle Three Buckets)** اللي بتختار النقط الأكثر تمثيلاً بصرياً.
- لما الجزء الظاهر أصغر من الحد، بنعرض كل النقط الخام (تفاصيل كاملة عند الزووم).

**النتيجة:** خط نضيف يحاكي الشكل، والمتصفح سلس مهما كان حجم الداتا.

---

## 3. الترند شكله "بحر"/بلوك أزرق صلب

**العرَض:** النظرة الشاملة بتطلع كتلة زرقا مالهاش معنى.

**السبب — جزئين:**
1. **الرسم:** `fill: true` (تعبئة تحت الخط) + نقط على كل عيّنة + انحناء (tension).
2. **الداتا نفسها:** بتتأرجح بكامل المدى (0↔1000) — ضوضاء مالهاش شكل أساسي.

**الحل:**
- في العرض التاريخي: **`fill = false`، `pointRadius = 0`، `tension = 0`، خط رفيع** → بدل الكتلة الصلبة بقى خط.
- استخدام LTTB بيحافظ على الشكل.
- **توضيح:** لو الداتا نفسها ضوضاء كاملة المدى، أي عرض أمين هيبان كشريط في النظرة البعيدة — الحل إنك **تزوّم** فتشوف التفاصيل، أو تستخدم بيانات حقيقية (مش random).

## 4. الـ Real-time chart متقطّع (مش بيرسم كل ثانية)

**العرَض:** التاج الثابت بيترسم نقطة كل ~3 ثواني بدل كل ثانية.

**السبب:** السيرفر بيبثّ التاج **بس لما يتغيّر** (deadband)، مع full refresh كل 3 ثواني. فالتشارت كان بيرسم على كل حدث، فالتاج الثابت بياخد نقطة كل 3 ثواني.

**الحل (زي ما الـ multitrend بيعمل):**
- فصلنا الرسم عن الأحداث: `updateChartData` بقى **يخزّن آخر قيمة بس**.
- `startRealtimeSampler()` يرسم نقطة كل `CHART_CONFIG.updateInterval` (1s) من آخر قيمة — إيقاع ثابت.
- أضفنا `setRealtimeInterval(ms)` للتحكّم في الإيقاع.

**النتيجة:** نقطة كل ثانية بالظبط للتاجات كلها، حتى الثابتة (خط أفقي بدل تقطيع).

## 5. تصدير الإكسل بيكراش / يهنّج الصفحة

**العرَض:**
```
Uncaught RangeError: Maximum call stack size exceeded at exportToXLSX
```

**السبب:** `Math.max(...numericValues)` — الـ spread (`...`) بيفرد كل عنصر كـ argument. مع 1.5 مليون عنصر بيتعدّى حد الـ call stack. وكمان: حتى لو اتصلّح، بناء sheet بمليون+ صف بيهنّج، وإكسل أصلاً أقصاه **1,048,576 صف/sheet**.

**الحل:**
- `computeStats()` تحسب min/max/avg في **loop واحد** بدل الـ spread (تشتغل لأي حجم).
- **حد أقصى للتصدير** `EXPORT_XLSX_MAX_ROWS = 1,048,575`؛ لو أكتر → رسالة بالإنجليزي تطلب تقليل النقط بدل ما يكراش.
- تحسين بناء الـ CSV (array + `join` بدل `+=`).
- إصلاح bug جانبي: القيمة `0` كانت بتتشال (`values[i] || ''`).

## 6. الحاجة لتعدّي 3 مليون نقطة (حجم النقل + الذاكرة)

**العرَض:** الـ JSON الضخم (~160 ميجا لـ 3 مليون) بطيء في النقل والـ `JSON.parse`، وبيستهلك ذاكرة مؤقتة كبيرة.

**الحل — Binary transport:**
- السيرفر يرجّع **Buffer** فيه `Float64` متتالية `[t, v, t, v, ...]` (16 بايت/نقطة) عند `?format=bin`.
- المتصفح يقرأ الـ `ArrayBuffer` كـ `Float64Array` مباشرة — **بدون `JSON.parse`** وبدون objects وسيطة.
- ~4× أصغر من JSON، وأسرع بكتير.
- رفعنا `MAX_RAW_ROWS` لـ 5,000,000.

**ملاحظة:** العائق الباقي هو ذاكرة Node على السيرف(بناء صفوف الـ recordset). للوصول لـ 10 مليون+ بثبات، الخطوة الجاية SQL streaming.

## 7. كل زووم كان بيبعت طلب جديد للسيرفر

**العرَض:** الزووم/التحريك كان بطيء لأنه بيطلب من السيرفر كل مرة.

**الحل:**
- **تحميل واحد لكل الخام في الرام** (binary).
- كل زووم/تحريك = **حساب محلي**: `lowerBound/upperBound` (binary search) لتحديد الجزء الظاهر + LTTB، بدون أي طلب.
- إيقاع تفاعلي فوري.

---

## 8. الكاش بيتراكم / مش بيتمسح

**العرَض:** قلق إن طلبات الـ history المتكررة تكدّس الذاكرة.

**الحل (مسح في حالتين):**
1. **عند كل Load Data:** بنمسح الكاش القديم **قبل** جلب الجديد (`fullData = []`)، فمفيش تراكم.
2. **عند إغلاق التشارت / تغيير الصفحة:** `closeChartModal` و `teardown` بيمسحوا كل الكاش ويوقفوا المؤقتات.
- أضفنا لوجز تأكيد في الكونسول (`cleared N cached points`).
- التأكد بالـ DevTools: **Performance Monitor → JS heap size**، أو **Memory → Heap snapshot (Comparison)** بعد إجبار الـ GC.

## 9. إخفاء تاج بيرجع يظهر عند الزووم (Multi-Trend History)

**العرَض:** تخفي تاج من الـ legend، وأول ما تزوّم/تتحرّك يرجع يظهر.

**السبب:** الزووم بيعيد بناء الـ datasets، فحالة الإخفاء (اللي Chart.js بيخزّنها في الـ meta) بتتصفّر.

**الحل:**
- نتتبّع حالة الإخفاء بنفسنا في `info.hidden` لكل تاج.
- `legend.onClick` مخصّص بيحفظ `info.hidden`.
- `buildDatasets()` بيطبّق `hidden: info.hidden` في كل إعادة رسم.

**النتيجة:** التاج المخفي يفضل مخفي عبر الزووم/التحريك لحد ما تظهره يدوياً.

---

## 10. إلغاء التحميل عند تغيير الصفحة أثناء الـ load

**العرَض:** لو قلبت صفحة أثناء تحميل multi-trend، العملية تفضل شغّالة والطلب مايتلغيش.

**الحل:**
- **`AbortController`** لكل fetch — `teardown` بيعمل `abort()` فيلغي الطلب الجاري (يبان `(canceled)` في الـ Network).
- **`loadToken`** بيتزوّد عند الإلغاء — اللوب المتسلسل بيتأكد من الـ token بعد كل `await` ويخرج بهدوء (مش بيحسبها "فشل").
- يمسح الكاش في نفس اللحظة.

## ملخص الدروس المستفادة

| الدرس | التطبيق |
|------|---------|
| الـ OFFSET pagination خطر على البيانات الكبيرة | query واحدة + index مناسب |
| الرسم محدود بالبكسلات مش بالداتا | decimation عند العرض (LTTB) |
| الـ spread (`...`) بيكسر على المصفوفات الضخمة | استخدم loops |
| JSON غالي على الملايين | binary transport |
| التفاعل السريع = البيانات محلية | كل الخام في الرام + حساب محلي |
| نظّف الذاكرة صراحةً | مسح عند Load + الإغلاق + AbortController |
